## ВВЕДЕНИЕ

Современная индустрия музыкального контента переживает период стремительной трансформации, обусловленной переходом от физических носителей и цифрового скачивания к стриминговой модели доставки контента. По данным аналитических агентств Midia Research и IFPI, доля стриминга в структуре мирового музыкального рынка превысила 65 процентов в 2024 году, а общая выручка от музыкального стриминга достигла 19,3 миллиарда долларов США. Количество активных пользователей музыкальных платформ превысило 600 миллионов человек, из которых более 220 миллионов являются платными подписчиками. Эта динамика обусловлена не только удобством доступа к огромному каталогу записей, но и формированием у пользователей ожидания персонализированного взаимодействия с музыкальным контентом.

Актуальность разработки информационных систем для работы с аудиоданными определяется несколькими ключевыми факторами. Возрастающий объём музыкальных каталогов — крупнейшие сервисы предлагают доступ к более чем 100 миллионам треков — делает ручную навигацию по каталогу неэффективной. Средний пользователь Spotify прослушивает менее 0,01% доступного каталога, что свидетельствует о критической необходимости интеллектуальных рекомендаций. Пользователи ожидают персонализированного взаимодействия: рекомендации, учитывающие контекст прослушивания — время суток, эмоциональное состояние, историю взаимодействий, — стали стандартом индустрии. Развитие технологий машинного обучения и цифровой обработки сигналов открывает новые возможности для анализа аудиоконтента, включая автоматическое определение жанра, настроения и тематики произведения. Однако существующие решения оставляют ряд пробелов: отсутствие семантического поиска по текстам песен, ограниченные возможности клиентского анализа аудио, недостаточная интеграция аудиоанализа в рекомендательные системы.

Объектом исследования является процесс организации потребления аудиоконтента в информационной среде. Предметом исследования выступают методы проектирования клиент-серверных информационных систем, алгоритмы рекомендаций на основе матричной факторизации, подходы к анализу аудиоданных с использованием Web API и методы семантического поиска по текстовому контенту.

Целью данной работы является разработка функционального прототипа информационной системы для аудиоданных, реализующей каталогизацию треков, управление плейлистами, рекомендации на основе гибридного алгоритма (коллаборативная фильтрация + матричная факторизация SVD + контентная фильтрация), анализ аудиохарактеристик средствами Web Audio API и семантический поиск по тематике текстов песен.

Для достижения поставленной цели необходимо решить следующие задачи:
- провести анализ существующих музыкальных стриминговых сервисов и выявить их ограничения;
- изучить теоретические основы рекомендательных систем и алгоритмы анализа аудиоданных;
- спроектировать архитектуру информационной системы, включая структуру базы данных и REST API;
- разработать серверную часть на базе Node.js и Express с использованием TypeScript;
- реализовать клиентскую часть на базе React с аудиоплеером и визуализацией;
- разработать гибридную рекомендательную систему на основе SVD и контентной фильтрации;
- реализовать модуль анализа аудиохарактеристик с использованием Web Audio API;
- разработать модуль семантического поиска по тематике текстов песен;
- провести тестирование разработанной системы и оценить её эффективность.

Методы исследования: анализ научной литературы и интернет-ресурсов, объектно-ориентированное проектирование, матричная факторизация (сингулярное разложение), алгоритмы цифровой обработки сигналов, методы естественной языковой обработки для семантического анализа текстов.

Методы анализа существующих сервисов включали: функциональный анализ (сравнение возможностей по 11 направлениям); анализ архитектурных решений (микросервисы, CDN, ABR); анализ алгоритмов рекомендаций (коллаборативная фильтрация, матричная因子изация, контентная фильтрация); анализ пользовательского опыта (навигация, визуальный дизайн, производительность).

Методы проектирования включали: структурный анализ (ER-диаграммы, диаграммы последовательности); проектирование REST API (ресурсная ориентация, HTTP-методы, статус-коды); проектирование алгоритмов (формулы, веса, пороговые значения); проектирование пользовательского интерфейса (мокапы, навигационные схемы).

Методы реализации включали: компонентное проектирование (React-компоненты, Express-маршруты, Sequelize-модели); тестирование (ручное, по сценариям); профилирование (измерение времени отклика, объёма хранилища).

Объём работы. Дипломная работа состоит из введения, четырёх глав, заключения и списка источников. Общий объём текста составляет приблизительно 18 000 слов. Работа содержит 3 таблицы, 11 листингов кода и 40 источников.

Структура работы. Дипломная работа состоит из введения, четырёх глав, заключения и списка источников. В первой главе проводится анализ существующих сервисов и излагаются теоретические основы рекомендательных систем и анализа аудиоданных. Во второй главе описывается проектирование системы: архитектура, база данных, REST API, алгоритмы рекомендаций и анализа. В третьей главе подробно рассматривается реализация каждого модуля с приведением ключевых фрагментов кода и описанием принципов работы. В четвёртой главе приводятся результаты тестирования и оценка эффективности системы.

## ГЛАВА 1. АНАЛИЗ СУЩЕСТВУЮЩИХ СЕРВИСОВ И ТЕОРЕТИЧЕСКИЕ ОСНОВЫ

### 1.1. Обзор рынка музыкальных стриминговых сервисов

Музыкальный стриминг — технология передачи аудиоконтента в реальном времени без предварительного скачивания файла на устройство пользователя. В отличие от классической модели загрузки, при которой весь файл передаётся до начала воспроизведения, стриминг реализует поэтапную доставку аудиоданных: воспроизведение начинается после заполнения небольшого буфера (обычно от 2 до 10 секунд), а последующие порции данных загружаются параллельно с прослушиванием. Технология основана на адаптивном битрейте (Adaptive Bitrate Streaming, ABR), который динамически регулирует качество аудиопотока в зависимости от пропускной способности канала связи.

Историческое развитие музыкального стриминга можно разделить на несколько этапов. Первый этап (2000–2008 годы) ознаменовался появлением сервиса Pandora, основанного на проекте Music Genome Project. Этот проект представлял собой масштабную базу данных, в которой каждый трек был описан посредством набора из более чем 450 музыкальных атрибутов — от темпа и тональности до инструментовки и вокальных характеристик. Рекомендации Pandora строились на сходстве этих атрибутов, что представляло собой один из первых коммерческих подходов к персонализации музыкального контента. В этот же период начали работу сервисы Rhapsody и Last.fm, последний из которых внедрил концепцию «таргетированного радио» на основе анализа поведения пользователя.

Второй этап (2008–2015 годы) был ознаменован запуском Spotify в 2008 году, который предложил модель freemium — бесплатный доступ с рекламой и платная подписка без рекламы с расширенным функционалом. Spotify стал первым сервисом,шедшим на массовый рынок с каталогом, насчитывающим десятки миллионов треков, и одновременно предлагающим качественные персонализированные рекомендации. В этот период также появились Apple Music (2015), Google Play Music (2011) и Яндекс.Музыка (2010), которая стала ведущим сервисом на российском рынке.

Третий этап (2015 – настоящее время) характеризуется доминированием алгоритмических рекомендаций и применением нейросетевых моделей. Spotify запустил функцию Discover Weekly, основанную на коллаборативной фильтрации и анализе аудио, Apple Music внедрил систему персонализированных плейлистов на основе поведенческого анализа, а Яндекс.Музыка развила нейросетевые алгоритмы для определения настроения и жанра. В это же время SoundCloud и Bandcamp обслуживают нишу независимых музыкантов, предлагая платформу для самостоятельной публикации контента.

Современный рынок музыкального стриминга характеризуется следующими показателями. Доля Spotify на мировом рынке составляет приблизительно 31 процент, Apple Music — 25 процентов, Amazon Music — 14 процентов, YouTube Music — 10 процентов, Tencent Music (включая QQ Music, Kugou и Kuwo) — 8 процентов. На российском рынке ведущие позиции занимает Яндекс.Музыка, доля которой оценивается в 45–50 процентов. Модели монетизации включают подписочные планы (от 5 до 15 долларов в месяц), бесплатный доступ с рекламой, семейные и студенческие тарифы, а также доходы от лицензирования для бизнеса.

### 1.2. Ключевые алгоритмы, используемые коммерческими сервисами

Детальный анализ алгоритмов рекомендаций, применяемых ведущими музыкальными стриминговыми платформами, позволяет выявить общие тенденции и уникальные решения.

**Spotify.** Рекомендательная система Spotify основана на комбинировании нескольких подходов. Алгоритм Discover Weekly, запущенный в 2015 году, использует коллаборативную фильтрацию (анализ взаимодействий «пользователь × трек»), модель Word2Vec для анализа «послушиваний» (trains of tracks — последовательности треков, которые пользователи слушали подряд), а также анализ аудиохарактеристик через собственные алгоритмы (energy, tempo, valence и т.д.). Нейросетевые модели применяются для генерации «аудиоэмбеддингов» — плотных векторных представлений треков, извлечённых из сырых аудиоданных. Модель Natural Language Processing (NLP) анализирует тексты блогов, обзоров и статей о музыке для определения «культурного контекста» трека. Все эти данные интегрируются в единую рекомендательную систему с использованием gradient-boosted деревьев.

**Apple Music.** Рекомендации Apple Music строятся на поведенческом анализе: система отслеживает жанровые предпочтения, частоту прослушиваний, время суток и контекст использования. Редакторские рекомендации (curated playlists) дополняют алгоритмические — команда музыкальных редакторов создаёт тематические плейлисты, которые интегрируются в персонализированные рекомендации. Apple Music Connect (прекращён в 2019) предоставлял социальную функциональность, но не оправдал ожиданий.

**Яндекс.Музыка.** Система рекомендаций Яндекс.Музыки использует нейросетевые модели для анализа аудио (определение жанра, настроения, инструментовки), коллаборативную фильтрацию на основе большого объёма данных о прослушиваниях пользователей Яндекса (поиск, карты, такси), а также контекстные факторы (время суток, погода, локация). Система «Настроение» автоматически группирует треки по эмоциональным категориям.

**SoundCloud.** Платформа использует рекомендации на основе: social graph (подписки на артистов); analytics (просмотров, лайков, репостов); тегов, добавленных авторами; «related tracks» — анализ потока прослушиваний для определения похожих треков.

**Tidal.** Рекомендации Tidal ориентированы на аудиофилов: система учитывает не только жанр и артиста, но и параметры качества (битрейт, формат кодирования), а также предпочтения в отношении «живых» и «студийных» записей.

### 1.3. Сравнительный анализ функциональных возможностей

Для выявления конкурентных преимуществ и пробелов существующих решений был проведён сравнительный анализ ведущих музыкальных стриминговых сервисов по одиннадцати ключевым направлениям функциональности.

Для выявления конкурентных преимуществ и пробелов существующих решений был проведён сравнительный анализ ведущих музыкальных стриминговых сервисов по одиннадцати ключевым направлениям функциональности.

**Потоковое воспроизведение аудио.** Все анализируемые сервисы (Spotify, Apple Music, Яндекс.Музыка, SoundCloud, Tidal) реализуют базовую функцию потокового воспроизведения. Spotify и Apple Music поддерживают адаптивный битрейт, автоматически регулирующий качество в зависимости от скорости интернет-соединения. Tidal предлагает HiFi-качество (FLAC, 1411 кбит/с) как ключевое конкурентное преимущество.

**Загрузка пользовательского контента.** Из перечисленных сервисов только SoundCloud и Bandcamp предоставляют пользователям возможность загружать собственные аудиозаписи. Ведущие коммерческие сервисы (Spotify, Apple Music) работают только с лицензированным контентом. Разрабатываемая система также предусматривает функцию загрузки треков, что расширяет её функциональность.

**Персонализированные рекомендации.** Все крупные сервисы реализуют персонализированные рекомендации. Spotify использует комбинацию коллаборативной фильтрации и анализа аудио («Audio Features»). Apple Music применяет поведенческий анализ с учётом жанровых предпочтений. Яндекс.Музыка использует нейросетевые модели. Однако ни один из анализируемых сервисов не реализует гибридную систему, комбинирующую SVD-факторизацию, контентную фильтрацию и анализ аудиохарактеристик в рамках единого алгоритма.

**Анализ аудиохарактеристик на стороне клиента.** Ни один из коммерческих сервисов не реализует анализ аудиохарактеристик непосредственно в клиентском приложении с использованием Web Audio API. Spotify анализирует аудио на серверной стороне с использованием собственных алгоритмов, а затем предоставляет предрассчитанные «Audio Features» (energy, valence, danceability, acousticness, tempo) через API. Разрабатываемая система реализует клиентский анализ в реальном времени, что позволяет определять характеристики для произвольных аудиозаписей.

**Семантический поиск по тематике текстов.** Ни один из анализируемых сервисов не предоставляет возможность поиска по смысловому содержанию текстов песен. Поиск в Spotify и Apple Music ограничен названиями треков, имёнами исполнителей и жанровыми метками. Разрабатываемая система реализует семантический поиск с определением 19 тематических категорий и 6 категорий настроения, что является уникальной функциональностью.

**Определение настроения.** Spotify и Яндекс.Музыка предлагают плейлисты, сгруппированные по настроению («Музыка для хорошего настроения», «Спокойные треки»). Однако автоматическое определение настроения на основе анализа текста песни — функция, отсутствующая в коммерческих решениях.

**Генерация плейлистов по настроению и времени суток.** Большинство сервисов предлагают статические редакторские плейлисты (например, «Discover Weekly» обновляется раз в неделю). Разрабатываемая система реализует динамическую генерацию плейлистов на основе текущего времени суток, выбранного настроения и жанра, с автоматическим подбором аудиохарактеристик.

**Статистика и аналитика для пользователя.** Spotify Wrapped и аналогичные функции предоставляют годовую статистику. Apple Music предоставляет базовую статистику прослушиваний. Разрабатываемая система предлагает детальную аналитику: распределение по жанрам, артистам, настроениям, темам, активность по часам и дням недели с интерактивными графиками.

**Социальные функции.** Spotify предлагает совместные плейлисты, обзор активности друзей и интеграцию с социальными сетями. Яндекс.Музыка поддерживает совместные плейлисты. SoundCloud реализует систему комментариев к трекам. Разрабатываемая система поддерживает публичные плейлисты и профили пользователей, но социальные функции не являются приоритетом.

**Качество аудио.** Spotify транслирует аудио в формате Ogg Vorbis с битрейтом от 96 до 320 кбит/с. Apple Music использует AAC 256 кбит/с. Tidal предлагает HiFi (FLAC 1411 кбит/с) и Master Quality (MQA). Яндекс.Музыка поддерживает битрейт до 320 кбит/с. Разрабатываемая система воспроизводит файлы в исходном формате без транскодирования.

**Радио и рекомендательные станции.** Все крупные сервисы предлагают функцию «радио» — бесконечный поток музыки, подобранный по предпочтениям пользователя. Разрабатываемая система реализует жанровое радио с рандомизированным подбором треков.

На основе проведённого анализа можно выделить следующие ключевые пробелы существующих решений, которые призвана устранить разрабатываемая система: отсутствие клиентского анализа аудиохарактеристик через Web Audio API; отсутствие семантического поиска по тематике текстов песен; отсутствие автоматического определения настроения на основе анализа текста; ограниченные возможности генерации плейлистов с учётом аудиохарактеристик.

### 1.3. Технологические особенности стриминговых платформ

Современные музыкальные стриминговые платформы характеризуются сложной распределённой архитектурой, обеспечивающей доставку аудиоконтента миллионам одновременных пользователей. В данном разделе рассматриваются ключевые технологические аспекты, определяющие функционирование таких систем.

**Архитектура серверной инфраструктуры.** Ведущие стриминговые сервисы используют микросервисную архитектуру, при которой каждый функциональный модуль (каталог, рекомендации, обработка платежей, аналитика) разворачивается как независимый сервис. Такая архитектура позволяет масштабировать отдельные компоненты независимо друг от друга. Spotify, например, эксплуатирует тысячи микросервисов, развёрнутых на базе Docker-контейнеров и оркестрируемых Kubernetes. Для хранения аудиофайлов используются объектные хранилища (совместимые с протоколом Amazon S3), обеспечивающие высокую доступность и горизонтальное масштабирование. Микросервисная архитектура предполагает использование шин сообщений (Apache Kafka, RabbitMQ) для асинхронного взаимодействия между сервисами, что обеспечивает отказоустойчивость и масштабируемость.

**Доставка контента.** Музыкальные файлы распределяются по глобальной сети доставки контента (Content Delivery Network, CDN), которая кэширует контент в географически распределённых узлах (point of presence, PoP). Это обеспечивает минимальную задержку передачи данных для пользователей из различных регионов. При обращении пользователя к треку запрос маршрутизируется к ближайшему CDN-узлу, что значительно сокращает время отклика по сравнению с обращением к центральному серверу. Крупнейшие CDN-провайдеры (Akamai, Cloudflare, Amazon CloudFront) имеют сотни узлов по всему миру, обеспечивая задержку менее 50 мс для большинства пользователей.

**Адаптивное битрейтное вещание (ABR).** Технология Adaptive Bitrate Streaming позволяет динамически регулировать качество аудиопотока в зависимости от текущей пропускной способности канала связи. Реализация ABR основана на сегментации аудиопотока на фрагменты продолжительностью от 2 до 10 секунд. Каждый фрагмент кодируется на нескольких уровнях качества (битрейта) — например, 96, 160, 256 и 320 кбит/с. Клиентское приложение непрерывно мониторит скорость загрузки и автоматически переключается на более высокий или низкий уровень качества, обеспечивая непрерывное воспроизведение без буферных пауз. Алгоритм выбора уровня качества учитывает: текущую скорость загрузки; размер буфера (оставшееся время воспроизведения); предыдущие колебания скорости; задержку переключения.

**Протоколы доставки.** Основными протоколами для стриминговой доставки аудио являются HTTP Live Streaming (HLS), разработанный Apple, и Dynamic Adaptive Streaming over HTTP (DASH), разработанный в рамках MPEG. HLS сегментирует поток на файлы .ts (Transport Stream) с метаданными .m3u8 (плейлист сегментов). DASH использует аналогичный подход с описанием в формате MPD (Media Presentation Description). Оба протокола работают поверх HTTP, что обеспечивает совместимость с существующей инфраструктурой CDN и файрволами. Различия: HLS поддерживает AES-128 шифрование сегментов; DASH поддерживает更多 форматы кодирования и DRM-системы.

**Форматы аудиокодирования.** Для потокового вещания применяются Lossy-форматы с различными компромиссами между качеством и размером файла. MP3 (MPEG-1 Audio Layer III) поддерживает битрейт от 32 до 320 кбит/с и остаётся наиболее совместимым форматом. AAC (Advanced Audio Coding) обеспечивает лучшее качество при том же битрейте по сравнению с MP3; используется Apple Music (256 кбит/с) и YouTube Music. Ogg Vorbis — открытый формат, используемый Spotify для снижения лицензионных отчислений (96–320 кбит/с). Для аудиофайлов высокого качества применяются форматы FLAC (Free Lossless Audio Codec, сжатие без потерь, обычно 800–1400 кбит/с) и MQA (Master Quality Authenticated, «студийное качество»). Spotify HiFi (обещанный, но не запущенный) планировал FLAC-стриминг.

**Буферизация и предсказательная загрузка.** Клиентские приложения реализуют буферизацию, при которой перед началом воспроизведения загружается определённый объём аудиоданных (буфер). Размер буфера определяет задержку между запросом и началом воспроизведения (typically 2–10 seconds), а также устойчивость к кратковременным колебаниям скорости сети. Современные системы используют предсказательную загрузку (prefetching), при которой загружаются не только текущий, но и последующие сегменты аудиопотока. Алгоритм prefetching анализирует паттерн прослушивания (последовательные треки в плейлисте) и предзагружает аудио для следующих треков.

**DRM (Digital Rights Management).** Музыкальные сервисы используют системы защиты авторских прав для предотвращения несанкционированного копирования. Основные DRM-системы: Widevine (Google), FairPlay Streaming (Apple), PlayReady (Microsoft). Шифрование применяется на уровне сегментов: ключ расшифровки передаётся через защищённый канал (license server), а сегменты шифруются AES-128.

Для прототипа разрабатываемой системы используется упрощённая архитектура, не предполагающая CDN, ABR и DRM, однако сохраняющая ключевые принципы: разделение клиентской и серверной частей, REST API для взаимодействия, формат JSON для передачи метаданных и прямая передача аудиофайлов по HTTP.

### 1.4. Теоретические основы рекомендательных систем

Рекомендательные системы (Recommendation Systems, RS) представляют собой класс информационных систем, предназначенных для прогнозирования предпочтений пользователя и предложения релевантных объектов. В контексте музыкальных стриминговых сервисов рекомендательные системы играют критическую роль, поскольку помогают пользователям обнаруживать новый контент среди миллионов доступных записей. В данном разделе рассматриваются основные подходы к построению рекомендательных систем, используемые в музыкальных платформах.

**Классификация рекомендательных систем.** Все методы рекомендаций можно разделить на три основные категории: коллаборативная фильтрация (Collaborative Filtering, CF), контентная фильтрация (Content-Based Filtering, CBF) и гибридные методы (Hybrid Methods), комбинирующие элементы первых двух подходов.

**Коллаборативная фильтрация** основана на предположении, что пользователи, имевшие схожее поведение в прошлом, будут иметь схожие предпочтения в будущем. Существует два основных подхода: user-based CF и item-based CF.

User-based CF строит рекомендации на основе анализа поведения «похожих» пользователей. Алгоритм работает следующим образом: для целевого пользователя находятся k наиболее близких пользователей на основе коэффициента корреляции Пирсона или косинусного сходства векторов оценок; далее формируются рекомендации на основе оценок, которые похожие пользователи поставили объектам, ещё не оценённым целевым пользователем. Коэффициент корреляции Пирсона вычисляется по формуле:

sim(u, v) = Σ[(r_ui − r̄_u)(r_vi − r̄_v)] / √[Σ(r_ui − r̄_u)² × Σ(r_vi − r̄_v)²],

где r_ui — оценка пользователя u объекту i, r̄_u — средняя оценка пользователя u.

Item-based CF строит рекомендации на основе сходства между объектами. Если пользователь взаимодействовал с объектом i, система рекомендует объекты, наиболее похожие на i, на основе совместных оценок пользователей. Этот подход более стабилен при больших каталогах, поскольку сходство между объектами меняется медленнее, чем между пользователями.

**Матричная факторизация (Matrix Factorization, MF)** является наиболее распространённым алгоритмом коллаборативной фильтрации в коммерческих системах. Идея состоит в приближении разреженной матрицы пользователь-объект произведением двух матриц меньшей размерности:

R ≈ P × Q^T,

где R — матрица оценок размерности m × n (m пользователей, n объектов), P — матрица пользовательских факторов размерности m × k, Q — матрица объектных факторов размерности n × k, k — количество скрытых факторов (latent factors).

Прогноз оценки для пары (пользователь u, объект i) вычисляется как:

r̂_ui = μ + p_u · q_i^T = μ + Σ(p_uf × q_if) для f от 1 до k,

где μ — общая средняя оценка, p_u — вектор пользовательского фактора, q_i — вектор объектного фактора.

Параметры P и Q обучаются методом градиентного спуска (Stochastic Gradient Descent, SGD) путём минимизации функции потерь:

L = Σ(r_ui − r̂_ui)² + λ(||p_u||² + ||q_i||²),

где λ — коэффициент регуляризации, предотвращающий переобучение. На каждой итерации градиентного спуска параметры обновляются:

p_uf := p_uf + lr × (e_ui × q_if − λ × p_uf),
q_if := q_if + lr × (e_ui × p_uf − λ × q_if),

где lr — скорость обучения (learning rate), e_ui = r_ui − r̂_ui — ошибка прогноза.

Существует также вариант Alternating Least Squares (ALS), в котором матрицы P и Q оптимизируются попеременно, фиксируя одну и решая задачу наименьших квадратов для другой. ALS удобен для распределённых вычислений и широко используется в Apache Spark MLlib.

**Контентная фильтрация (Content-Based Filtering)** строит рекомендации на основе анализа свойств объектов и предпочтений пользователя. Для музыкальных треков в качестве признаков используются: жанр, исполнитель, аудиохарактеристики (energy, valence, danceability, acousticness, tempo), текст песни, тематика и настроение.

Профиль пользователя строится как вектор средних значений аудиохарактеристик听过 треков, которым пользователь поставил «лайк» или которые он много слушал. Рекомендация формируется на основе косинусного сходства между вектором аудиохарактеристик кандидата и вектором профиля пользователя:

cosine_sim(A, B) = (A · B) / (||A|| × ||B||),

где A — вектор аудиохарактеристик кандидата, B — вектор профиля пользователя.

Контентная фильтрация обладает рядом преимуществ: не требует данных о поведении других пользователей (решает проблему холодного старта для новых объектов); обеспечивает прозрачность рекомендаций (пользователь может понять, почему объект рекомендован); позволяет учитывать контекст (время суток, настроение). Основной недостаток — ограниченность каталога: система может рекомендовать только объекты, похожие на уже известные пользователю, затрудняя обнаружение нового контента (проблема «малого мира»).

**Нейросетевые подходы.** В последние годы получили распространение нейросетевые модели для рекомендаций. Автоэнкодеры (Autoencoders) обучаются реконструировать вектор оценок пользователя, а скрытое представление используется для генерации рекомендаций. Нейросетевые коллаборативные фильтры (Neural Collaborative Filtering, NCF) заменяют скалярное произведение в классическом MF на нелинейную функцию, аппроксимированную многослойным персептроном. Модели на основе рекуррентных нейронных сетей (RNN, LSTM) учитывают последовательность прослушиваний, моделируя эволюцию предпочтений пользователя во времени. Трансформерные архитектуры (BERT4Rec, SASRec) применяют механизм внимания для моделирования долгосрочных зависимостей в истории прослушиваний.

**Гибридные методы** комбинируют преимущества коллаборативной и контентной фильтрации. Существует несколько стратегий комбинирования: взвешенное (каждый метод вносит вклад с определённым весом), каскадное (сначала применяется один метод, затем результат фильтруется вторым), мета-learning (обучается модель, которая определяет, какой метод использовать в каждом конкретном случае). Коммерческие сервисы, как правило, используют гибридные подходы: Spotify комбинирует коллаборативную фильтрацию с анализом аудио; Apple Music — поведенческий анализ с редакторскими рекомендациями; Яндекс.Музыка — нейросетевые модели с контекстными факторами.

**Проблемы рекомендательных систем.** Холодный старт (cold start) — проблема рекомендаций для новых пользователей (мало данных о предпочтениях) и новых объектов (мало взаимодействий). Существует несколько подходов к решению: использование демографических данных (возраст, пол, регион) для формирования начального профиля; предложение «стартовых» плейлистов на основе популярных треков; использование контентной фильтрации, не зависящей от истории взаимодействий.

Разреженность матрицы (sparsity) — в реальных системах большинство пользователей взаимодействует лишь с малой долей каталога. В Spotify, например, при каталоге из 100 миллионов треков средний пользователь прослушивает менее 10 тысяч (0,01%), что делает матрицу взаимодействий более чем на 99,99% разреженной. Матричная факторизация успешно справляется с этой проблемой, поскольку проектирует разреженную матрицу в плотное低-dimensional пространство.

Пузырь фильтров (filter bubble) — проблема, при которой система рекомендует только контент, подтверждающий существующие предпочтения пользователя, ограничивая разнообразие. Музыкальные сервисы борются с этой проблемой через: добавление «случайности» в рекомендации (epsilon-greedy стратегия); введение «исследовательского» компонента (explore vs exploit); редакторские плейлисты, формируемые людьми.

Масштабируемость — необходимость обработки миллиардов взаимодействий в реальном времени. Для решения этой проблемы используются: распределённые вычисления (Apache Spark, Hadoop); онлайн-обновление моделей (incremental learning); кэширование результатов; приближённые алгоритмы (approximate nearest neighbors).

**Нейросетевые модели для музыкальных рекомендаций.** Автоэнкодеры (Variational Autoencoders, VAE) обучаются реконструировать вектор оценок пользователя. Архитектура: энкодер сжимает вектор оценок в латентное пространство (hidden layer), декодер восстанавливает вектор. Латентное представление используется для генерации рекомендаций. Модель Mult-VAE (Liang et al., 2018) показала state-of-the-art результаты на датасетах音乐推荐.

Нейросетевые коллаборативные фильтры (NCF, He et al., 2017) заменяют скалярное произведение в классическом MF на нелинейную функцию, аппроксимированную многослойным персептроном. Архитектура GMF (Generalized Matrix Factorization) комбинирует линейную (MF) и нелинейную (MLP) компоненты.

Модели на основе рекуррентных нейронных сетей (RNN, GRU, LSTM) учитывают последовательность прослушиваний, моделируя эволюцию предпочтений пользователя во времени. Модель GRU4Rec (Hidasi et al., 2016) использовала GRU-сети для предсказания следующего трека в сессии прослушивания.

Трансформерные архитектуры (BERT4Rec, SASRec) применяют механизм self-attention для моделирования долгосрочных зависимостей в истории прослушиваний. BERT4Rec (Sun et al., 2019) использует双向 attention (как BERT) для учёта контекста из прошлого и будущего в последовательности.

Graph Neural Networks (GNN) для рекомендаций представляют взаимодействия «пользователь × трек» как граф и используют message passing для агрегации информации из соседних узлов. Модель LightGCN (He et al., 2020) показала значительное улучшение по сравнению с классическим MF на датасетах MovieLens и Amazon.

### 1.5. Анализ алгоритмов анализа аудиоданных

Анализ аудиоданных представляет собой совокупность методов цифровой обработки сигналов (Digital Signal Processing, DSP), направленных на извлечение осмысленных характеристик из сырых аудиоданных. В контексте музыкальных рекомендательных систем анализ аудио позволяет автоматически определять жанр, настроение, энергичность и другие характеристики трека, что может использоваться как для построения профиля пользователя, так и для кластеризации каталога.

**Предварительная обработка аудиосигнала.** Исходный аудиосигнал, представленный в формате PCM (Pulse Code Modulation), характеризуется высокой частотой дискретизации (обычно 44100 или 48000 Гц) и большим объёмом данных. Для аналитических вычислений выполняется даунсэмплирование (downsampling) — уменьшение частоты дискретизации до 8000 Гц без существенной потери аналитической информации о среднечастотных и низкочастотных компонентах, определяющих музыкальные характеристики.

**Среднеквадратичное значение (RMS).** RMS (Root Mean Square) характеризует среднюю энергию аудиосигнала и вычисляется как:

RMS = √[(1/N) × Σ(x_i²)],

где x_i — отсчёты аудиосигнала, N — количество отсчётов. Значение RMS нормализуется к диапазону [0, 1] и используется как оценка энергии (energy) трека. Высокие значения RMS соответствуют громким, энергичным композициям, низкие — тихим, акустическим. RMS является одной из наиболее стабильных и широко используемых характеристик в музыкальном анализе, поскольку коррелирует с воспринимаемой громкостью и «плотностью» микса.

**Частота пересечения нуля (Zero-Crossing Rate, ZCR).** ZCR определяет количество раз, когда аудиосигнал меняет знак (проходит через ноль), и вычисляется как:

ZCR = (1/N) × Σ|sgn(x_i) − sgn(x_{i−1})|,

где sgn — функция знака. Высокие значения ZCR характерны для звонких, высокочастотных звуков (например, хай-хэта в ударной установке), низкие — для глубоких, низкочастотных звуков. ZCR используется как компонент при вычислении «валентности» (valence) — показателя, характеризующего эмоциональную тональность трека. ZCR также используется для различения гармонических и перкуссивных звуков: гармонические звуки (вокал, гитара) имеют более высокий ZCR, чем перкуссивные (ударные).

**Спектральный анализ.** Спектральный центроид (Spectral Centroid) определяет «центр масс» спектра сигнала и характеризует воспринимаемую яркость звука. Вычисляется как взвешенное среднее частотных компонентов, где весами являются амплитуды соответствующих частот:

SC = Σ(f_k × |X_k|) / Σ|X_k|,

где f_k — частота k-го бина, |X_k| — амплитуда k-го бина в спектре Фурье. Высокие значения спектрального центроида соответствуют «яркому», «звонкому» звуку (электрогитара, синтезатор), низкие — «тёмному», «глухому» (бас-гитара, контрабас).

Спектральное рассеяние (Spectral Spread) характеризует ширину спектрального распределения энергии и вычисляется как стандартное отклонение спектра относительно центроида:

SS = √[Σ((f_k − SC)² × |X_k|) / Σ|X_k|].

Высокое спектральное рассеяние соответствует «широкому» звуку (оркестр, хор), низкое — «узкому» (сольный инструмент, моно-запись). Спектральная плоскость (Spectral Flatness) определяет, насколько равномерно распределена энергия по частотам: высокая плоскость соответствует шумоподобным сигналам (ударные, тарелки), низкая — тональным (вокал, пианино).

**Onset Detection (определение моментов начала нот).** Метод определения моментов начала нот (onsets) основан на анализе изменений энергии сигнала во времени. Сигнал разбивается на короткие блоки (окна) продолжительностью 20–50 мс, для каждого блока вычисляется RMS. Резкое увеличение RMS (более чем на пороговое значение, например 0.015) по сравнению с предыдущим блоком интерпретируется как начало новой ноты (onset). Частота onset'ов (onset rate) используется как мера «танцевальности» (danceability) — треки с регулярными, чёткими ритмическими акцентами (ударные, бас) имеют высокий onset rate, в то время как плавные, легато-фрагменты (струнные, вокал) — низкий.

Для повышения точности используется полуволновая ректификация (half-wave rectification) разности RMS: diff[i] = max(0, RMS[i+1] − RMS[i]), что исключает отрицательные изменения энергии (затухание нот) из расчёта.

**Определение темпа (BPM).** Темп (Beats Per Minute) определяется методом автокорреляции огибающей амплитуды (amplitude envelope). Алгоритм: сигнал разбивается на окна по 2 секунды с шагом (hop size) в четверть окна; для каждого окна вычисляется RMS, формируя последовательность значений огибаенной; далее вычисляется автокорреляция огибаенной для диапазона лагов, соответствующих диапазону темпов от 60 до 200 BPM:

min_lag = (60 / max_BPM) × (sampleRate / hopSize),
max_lag = (60 / min_BPM) × (sampleRate / hopSize).

Лаг с максимальной корреляцией определяет период ритма, из которого вычисляется BPM:

BPM = 60 / (best_lag × hop_size / sample_rate).

Для повышения точности используется полуволновая ректификация огибающей перед вычислением автокорреляции, что усиливает периодические пики и снижает влияние шума.

**Извлечение мел-кривой (Mel-frequency cepstral coefficients, MFCC).** MFCC представляют собой набор из 13–20 коэффициентов, описывающих форму спектра сигнала в логарифмической шкале мел. Алгоритм: преобразование Фурье (FFT) для получения спектра; применение фильтровой банки мел (20–40 треугольных фильтров, равномерно распределённых по шкале мел); логарифмирование энергии в каждом фильтре; обратное дискретное косинусное преобразование (DCT) для получения коэффициентов. MFCC широко используются в задачах распознавания речи и классификации музыки, поскольку эффективно кодируют спектральную форму с низкой размерностью.

**Хромаграмма (Chroma Features).** Хромаграмма представляет распределение энергии по 12 музыкальным нотам (C, C#, D, ..., B) независимо от октавы. Алгоритм: разбиение спектра FFT на 12 бинов, соответствующих нотам; суммирование энергии во всех октавах для каждой ноты; нормализация. Хромаграмма эффективна для определения тональности (мажор/минор), гармонической прогрессии и ритмических паттернов.

**Web Audio API как инструмент клиентского анализа.** Web Audio API — интерфейс браузера, предоставляющий программный доступ к аудиоподсистеме. Основные компоненты: AudioContext — контекст для управления аудио; AudioBufferSourceNode — источник аудиоданных; AnalyserNode — узел для получения частотных данных в реальном времени; GainNode — узел регулировки громкости; BiquadFilterNode — фильтр (низкочастотный, высокочастотный, полосовой); ConvolverNode — реверберация; OscillatorNode — генератор тонов. Для анализа аудиофайлов используется метод decodeAudioData(), который декодирует аудиоданные в формате ArrayBuffer в объект AudioBuffer с доступом к сырым PCM-данным через метод getChannelData(). Преимущество клиентского анализа заключается в отсутствии необходимости передачи аудиоданных на сервер, что обеспечивает конфиденциальность и снижает нагрузку на серверную инфраструктуру.

### 1.6. Цели и задачи разработки

На основе проведённого анализа существующих музыкальных стриминговых сервисов и теоретических основ были выявлены следующие ключевые пробелы, которые призвана устранить разрабатываемая система:

1. Отсутствие у коммерческих сервисов функции клиентского анализа аудиохарактеристик через Web Audio API, что не позволяет определять характеристики для произвольных аудиозаписей.
2. Отсутствие семантического поиска по тематике текстов песен в существующих решениях.
3. Ограниченные возможности автоматического определения настроения на основе анализа текста песни.
4. Отсутствие гибридной системы рекомендаций, комбинирующей SVD-факторизацию, контентную фильтрацию и анализ аудио в рамках единого алгоритма.

Целью разработки является создание функционального прототипа информационной системы для аудиоданных (MusicStream), реализующей перечисленные функциональные возможности.

Задачи разработки:
- провести функциональный анализ существующих сервисов и определить направления совершенствования;
- спроектировать клиент-серверную архитектуру с трёхуровневой моделью (клиент – сервер – база данных);
- разработать реляционную модель данных для хранения информации о треках, пользователях, плейлистах и взаимодействиях;
- реализовать REST API с JWT-аутентификацией и ролевой моделью (пользователь, артист, администратор);
- разработать гибридную рекомендательную систему на основе SVD, контентной и коллаборативной фильтрации;
- реализовать модуль анализа аудиохарактеристик с использованием Web Audio API на стороне клиента;
- разработать модуль семантического поиска по тематике текстов песен;
- реализовать генератор плейлистов с учётом настроения и аудиохарактеристик;
- разработать пользовательский интерфейс с аудиоплеером, визуализацией и статистикой;
- провести тестирование системы и оценить эффективность реализованных алгоритмов.

## ГЛАВА 2. ПРОЕКТИРОВАНИЕ ИНФОРМАЦИОННОЙ СИСТЕМЫ

### 2.1. Общая архитектура системы

Разрабатываемая информационная система MusicStream спроектирована на основе клиент-серверной архитектуры с чётким разделением ответственности между компонентами. Архитектура системы соответствует трёхуровневой модели: клиентский уровень (представление), серверный уровень (бизнес-логика) и уровень хранения данных.

**Клиентский уровень** реализован на базе фреймворка React и отвечает за визуализацию пользовательского интерфейса, взаимодействие пользователя с системой и воспроизведение аудиоконтента. Клиентское приложение не содержит бизнес-логики — все вычислительные операции (рекомендации, анализ, поиск) выполняются на сервере. Исключением является анализ аудиохарактеристик, который выполняется на стороне клиента с использованием Web Audio API, что позволяет анализировать аудиофайлы без передачи сырых данных на сервер.

**Серверный уровень** реализован на базе платформы Node.js с фреймворком Express и отвечает за обработку запросов, бизнес-логику, взаимодействие с базой данных и предоставление REST API. Серверная часть написана на языке TypeScript, что обеспечивает строгую типизацию и снижает количество ошибок на этапе разработки. Серверный уровень включает модули аутентификации, управления треками, рекомендательной системы, анализа тематики текстов и статистики.

**Уровень хранения данных** использует реляционную базу данных SQLite с ORM-прослойкой Sequelize. Выбор SQLite обусловлен портативностью (база данных представлена одним файлом) и отсутствием необходимости в отдельном сервере БД, что упрощает развёртывание прототипа. Sequelize обеспечивает абстракцию над SQL-запросами, автоматическую миграцию схем и удобный доступ к данным через объектно-ориентированный интерфейс. Для промышленной эксплуатации рекомендуется переход на PostgreSQL или MySQL, которые обеспечивают более высокую производительность при большом объёме данных и поддержку сетевых подключений.

Взаимодействие между уровнями осуществляется по протоколу HTTP через REST API. Клиент отправляет HTTP-запросы (GET, POST, PUT, DELETE) к серверным маршрутам, сервер обрабатывает запросы и возвращает ответ в формате JSON. Аутентификация реализована на основе JSON Web Token (JWT): клиент передаёт токен в заголовке Authorization каждого запроса, сервер проверяет токен и идентифицирует пользователя. Формат ответа API единообразен: успешные ответы содержат данные (объект или массив), ошибки — объект с полем error (строка с описанием).

### 2.2. Проектирование базы данных

Реляционная модель данных включает одиннадцать таблиц, связанных отношениями «один-ко-многим» и «многие-ко-многим». Ниже описывается структура каждой таблицы и используемые типы данных.

**Таблица users** хранит информацию о пользователях системы. Поля: id (UUID, первичный ключ), username (строка до 50 символов, уникальный), email (строка до 255 символов, уникальный, проверка формата), password (строка до 255 символов — хеш пароля), displayName (строка до 100 символов — отображаемое имя), avatar (строка до 500 символов, nullable — путь к файлу аватара), role (перечисление: user, admin, artist), bio (текст, nullable — описание профиля), createdAt, updatedAt (метки времени).

**Таблица artists** хранит информацию об исполнителях. Поля: id (UUID), name (строка до 200 символов), bio (текст, nullable), image (строка до 500 символов, nullable — путь к изображению), verified (булево — верифицированный исполнитель), createdAt, updatedAt.

**Таблица genres** хранит музыкальные жанры. Поля: id (UUID), name (строка до 100 символов), slug (строка до 100 символов, уникальный — URL-идентификатор), image (строка до 500 символов, nullable).

**Таблица albums** хранит информацию об альбомах. Поля: id (UUID), title (строка до 300 символов), artistId (UUID, внешний ключ к artists), coverUrl (строка до 500 символов, nullable), year (целое, nullable), type (перечисление: album, single, ep), createdAt, updatedAt.

**Таблица tracks** является центральной таблицей системы. Поля: id (UUID), title (строка до 300 символов), artistId (UUID, внешний ключ к artists), albumId (UUID, nullable, внешний ключ к albums), genreId (UUID, nullable, внешний ключ к genres), duration (целое — длительность в секундах), filePath (строка до 500 символов — путь к аудиофайлу), coverUrl (строка до 500 символов, nullable), lyrics (текст, nullable — текст песни), themes (текст, по умолчанию «[]» — JSON-массив тематических меток), mood (строка до 100 символов, nullable — определённое настроение), plays (целое, по умолчанию 0 — счётчик прослушиваний), likes (целое, по умолчанию 0 — счётчик лайков), uploadedBy (UUID, внешний ключ к users), explicit (булево — метка 18+), tags (текст, по умолчанию «[]» — JSON-массив тегов), tempo (вещественное, nullable — темп BPM), energy (вещественное, nullable — энергия 0–1), valence (вещественное, nullable — валентность 0–1), acousticness (вещественное, nullable — акустичность 0–1), danceability (вещественное, nullable — танцевальность 0–1), createdAt, updatedAt.

**Таблица playlists** хранит информацию о плейлистах. Поля: id (UUID), name (строка до 200 символов), description (текст, nullable), coverUrl (строка до 500 символов, nullable), userId (UUID, внешний ключ к users), isPublic (булево, по умолчанию true), isCollaborative (булево, по умолчанию false), createdAt, updatedAt.

**Таблица playlist_tracks** — промежуточная таблица для связи «многие-ко-многим» между playlists и tracks. Поля: playlistId (UUID, внешний ключ), trackId (UUID, внешний ключ), position (целое — позиция трека в плейлисте).

**Таблица likes** — промежуточная таблица для связи «многие-ко-многим» между users и tracks (лайки). Поля: userId (UUID, внешний ключ), trackId (UUID, внешний ключ).

**Таблица play_history** хранит историю прослушиваний. Поля: id (UUID), userId (UUID, внешний ключ к users), trackId (UUID, внешний ключ к tracks), playedAt (время — момент прослушивания), progress (целое, по умолчанию 0 — прогресс прослушивания в секундах).

**Таблица track_genres** — промежуточная таблица для связи «многие-ко-многим» между tracks и genres (один трек может принадлежать нескольким жанрам). Поля: trackId (UUID, внешний ключ), genreId (UUID, внешний ключ).

Описанные связи обеспечивают целостность данных и позволяют эффективно выполнять запросы, связывающие информацию о пользователях, треках, плейлистах и взаимодействиях.

### 2.4. Проектирование рекомендательной системы

Рекомендательная система MusicStream реализует гибридный подход, комбинирующий три метода: контентную фильтрацию, коллаборативную фильтрацию и матричную.factorизацию SVD. Выбор гибридного подхода обусловлен необходимостью преодоления ограничений каждого из методов по отдельности: контентная фильтрация не учитывает поведение других пользователей; коллаборативная фильтрация страдает от холодного старта; SVD-модель требует достаточного объёма данных для обучения.

Итоговая оценка кандидата вычисляется как взвешенная комбинация оценок трёх методов:

hybrid_score = content_score × 0.4 + collab_score × 0.2 + svd_score × 4.

Коэффициенты весов подобраны эмпирически с учётом следующих факторов: контентная фильтрация получает наибольший вес (0.4), поскольку обеспечивает основную персонализацию на основе аудиохарактеристик и жанровых предпочтений; SVD получает мультипликативный коэффициент 4, поскольку его предсказания имеют меньший масштаб (средние значения близки к 0, отклонение ±1–2), в то время как контентная и коллаборативная оценки могут достигать десятков; коллаборативная фильтрация получает вес 0.2, дополняя рекомендации данными о поведении других пользователей.

**Построение профиля пользователя.** Профиль пользователя строится на основе анализа двух источников данных: лайков (таблица likes) и истории прослушиваний (таблица play_history). Лайки получают вес ×3 (сильная положительная обратная связь: пользователь сознательно отметил трек как понравившийся); прослушивания — вес ×1 (умеренная обратная связь: пользователь слушал трек, но не обязательно намеренно). Профиль включает:
- genrePreferences — Map<genreId, weight>: предпочтения по жанрам (суммарный вес лайков и прослушиваний для каждого жанра);
- artistPreferences — Map<artistId, weight>: предпочтения по артистам;
- avgEnergy, avgValence, avgDanceability, avgAcousticness, avgTempo — средние аудиохарактеристики, вычисленные как среднее арифметическое соответствующих характеристик всех лайкнутых и прослушанных треков;
- likedTrackIds — Set идентификаторов лайкнутых треков (для исключения из рекомендаций);
- recentlyPlayedIds — Set идентификаторов недавно прослушанных треков (для исключения из рекомендаций).

Для исключения уже известных треков из рекомендаций используется объединение likedTrackIds и recentlyPlayedIds. Это гарантирует, что система не будет рекомендовать треки, которые пользователь уже оценил или недавно слушал.

**Контентная фильтрация.** Оценка контентной фильтрации (content_score) вычисляется для каждого кандидата на основе трёх компонент: жанровое сходство (вес ×2), артистическое сходство (вес ×3) и сходство аудиохарактеристик ( вес ×5). Сходство аудиохарактеристик определяется косинусным расстоянием между пятимерными векторами кандидата и профиля пользователя:

trackVector = [energy, valence, danceability, acousticness, tempo/200],
userVector = [avgEnergy, avgValence, avgDanceability, avgAcousticness, avgTempo/200],
featureSimilarity = cosineSimilarity(trackVector, userVector).

Нормализация темпа к диапазону [0, 1] (деление на 200) обеспечивает совместимость масштабов при вычислении косинусного сходства: темп 120 BPM становится 0.6, темп 80 BPM — 0.4, что делает его сопоставимым с другими характеристиками (energy 0–1, valence 0–1 и т.д.).

Дополнительно учитывается популярность трека: score += log1p(plays) × 0.5 + log1p(likes) × 0.3. Логарифмическое штрафование (log1p) предотвращает доминирование сверхпопулярных треков: трек с 1000 прослушиваниями получает бонус log1p(1000) × 0.5 ≈ 3.45, а трек с 100000 прослушиваниями — log1p(100000) × 0.5 ≈ 5.76, что обеспечивает более справедливое ранжирование.

**Коллаборативная фильтрация.** Оценка коллаборативной фильтрации (collab_score) строится на основе анализа поведения пользователей, которые также прослушивали данный трек. Алгоритм: находятся пользователи, прослушивавшие трек-кандидат (не включая целевого пользователя), с лимитом 50 записей; анализируются их взаимодействия (с лимитом 200 записей) для определения треков, которые они также слушали; оценка формируется из трёх компонент: логарифм популярности трека среди похожих пользователей (log1p(trackPlayCounts) × 2), количество лайков от похожих пользователей (similarUserLikes.length × 1.5), доля похожих пользователей от общего числа (userIds.length / 20 × 3). Последний компонент обеспечивает приоритет треков, которые нравятся широкому кругу пользователей.

**SVD-модель.** Матричная因子изация реализует алгоритм SVD (Singular Value Decomposition) для предсказания оценок. Модель обучается на матрице взаимодействий «пользователь × трек», где оценки формируются следующим образом: каждое прослушивание начисляет 1 балл; повторные прослушивания увеличивают оценку на 0.5 (максимум 5), что отражает нарастающий интерес; лайк начисляет 3 балла; комбинация прослушивания и лайка суммируется (максимум 5). Минимальное количество взаимодействий для обучения — 5 записей; при меньшем объёме данных модель не строится, и предсказание возвращает общую среднюю оценку.

Матрица оценок передаётся в алгоритм trainSVD с параметрами: количество скрытых因子ов k = 20 (или min(users, items) − 1, если пользователей/треков menos 21); скорость обучения lr = 0.005; регуляризация λ = 0.02; количество итераций = 50. Алгоритм минимизирует функцию потерь:

L = Σ(r_ui − (μ + p_u · q_i))² + λ(||p_u||² + ||q_i||²),

где μ — общая средняя оценка. Матрицы P и Q инициализируются случайными значениями из диапазона [−0.05, 0.05] и обновляются на каждой итерации градиентным спуском:

p_uf := p_uf + lr × (error × q_if − λ × p_uf),
q_if := q_if + lr × error × p_uf − λ × q_if).

Для повышения производительности модель кэшируется с временем жизни (TTL) 5 минут. Кэширование реализовано через глобальные переменные cachedSVDModel и svdModelTimestamp; при каждом запросе рекомендаций проверяется актуальность кэша.

**Подобные треки.** Функция getSimilarTracks определяет треки, похожие на заданный. Алгоритм: загрузка целевого трека со связанными данными (Artist, Genre); формирование вектора аудиохарактеристик; фильтрация кандидатов по жанру или артисту (Op.or); вычисление косинусного сходства для каждого кандидата; добавление бонусов за совпадение артиста (+5) и жанра (+2); учёт популярности (log1p(plays) × 0.3); сортировка и возврат топ-N.

### 2.5. Проектирование модуля анализа аудио

Модуль анализа аудиохарактеристик разработан на стороне клиента с использованием Web Audio API и предназначен для автоматического определения пяти ключевых характеристик трека: energy (энергия), valence (валентность/эмоциональная тональность), danceability (танцевальность), acousticness (акустичность) и tempo (темп в BPM).

**Поток данных.** Аудиофайл загружается с сервера через fetch, декодируется методом AudioContext.decodeAudioData() в объект AudioBuffer. Из AudioBuffer извлекаются сырые PCM-данные канала (getChannelData(0)) и частота дискретизации. Далее выполняется даунсэмплирование до частоты дискретизации 8000 Гц для уменьшения объёма вычислений.

**Вычисление energy.** Энергия определяется через RMS (Root Mean Square) аудиосигнала. RMS нормализуется к диапазону [0, 1] путём умножения на коэффициент 3.5 и ограничения сверху единицей. Высокие значения RMS соответствуют громким, плотным миксам (метал, электронная музыка), низкие — тихим, акустическим записям.

**Вычисление valence.** Валентность (эмоциональная тональность) определяется как комбинация двух характеристик: спектрального центроида и частоты пересечения нуля. Спектральный центроид вычисляется для блоков по 1024 отсчёта как взвешенное среднее оценки центроида (умножение ZCR на частоту дискретизации и деление на 2) с весами RMS каждого блока. Яркость (brightness) нормализуется делением на 4000. Итоговая valence = min(1, brightness × 0.7 + ZCR × 2).

**Вычисление danceability.** Танцевальность определяется через анализ onset'ов (моментов начала нот). Аудиосигнал разбивается на блоки по 50 мс, для каждого блока вычисляется RMS. Рост RMS более чем на 0.015 по сравнению с предыдущим блоком интерпретируется как onset. Onset rate (отношение количества onset'ов к общему числу блоков) нормализуется к диапазону [0, 1]. Danceability = min(1, onsetRate × 3 + rms × 0.5).

**Вычисление acousticness.** Акустичность определяется через анализ низкочастотных и высокочастотных компонент. Сигнал разбивается на блоки по 20 мс; чётные блоки суммируются как «низкочастотные», нечётные — как «высокочастотные». Доля низкочастотной энергии (lowRatio) используется как основная мера акустичности. Дополнительно учитывается «плоскость» спектра (flatness), определяемая отклонением ZCR от 0.5. Acousticness = min(1, lowRatio × 0.6 + (1 − rms) × 0.3 + flatness × 0.1).

**Вычисление tempo (BPM).** Темп определяется методом автокорреляции огибающей амплитуды. Сигнал разбивается на окна по 2 секунды с шагом в четверть окна (hop size = window / 4). Для каждого окна вычисляется RMS, формируя последовательность значений огибающей. Далее вычисляется разность последовательных значений огибающей (diff) с ограничением неотрицательных значений (half-wave rectification). Автокорреляция diff вычисляется для диапазона лагов, соответствующего BPM от 60 до 200. Лаг с максимальной корреляцией определяет период ритма, из которого вычисляется BPM = 60 / (bestLag × hopSize / sampleRate).

**Интеграция с сервером.** После определения аудиохарактеристик на клиенте выполняется PUT-запрос к серверу для сохранения характеристик в базе данных (PUT /api/tracks/:id/features). В дальнейшем характеристики загружаются вместе с метаданными трека, что исключает необходимость повторного анализа. При воспроизведении трека клиент сначала проверяет, есть ли уже сохранённые характеристики (energy != null); если да — использует их; если нет — выполняет клиентский анализ. Результаты анализа кэшируются на клиенте в Set (analyzedTracksRef) с ограничением размера 200 записей для предотвращения утечки памяти.

**Преимущества клиентского анализа.** Клиентский подход позволяет: анализировать произвольные аудиофайлы без серверной обработки; обеспечивать конфиденциальность (аудиоданные не покидают устройство пользователя); снижать нагрузку на сервер; работать офлайн (после первичной загрузки файла). Ограничения: производительность зависит от устройства пользователя; Web Audio API поддерживается не во всех браузерах; точность ниже по сравнению с серверными алгоритмами ( librosa, Essentia ).

### 2.6. Проектирование семантического поиска

Модуль семантического поиска реализует поиск по тематике текстов песен, основанный на анализе ключевых слов, определении тематических категорий и вычислении семантического сходства. В отличие от классического полнотекстового поиска, который ищет точные совпадения подстроки в названиях и метаданных, семантический поиск анализирует смысловое содержание текста песни.

**Анализ тематики текста.** Текст песни анализируется на наличие ключевых слов, сгруппированных по 19 тематическим категориям: любовь (love), разлука (breakup), дружба (friendship), грусть (sadness), радость (joy), жизнь (life), свобода (freedom), город (city), природа (nature), ностальгия (nostalgия), меланхолия (melancholy), надежда (hope), тревога (anxiety), одиночество (loneliness), страсть (passion), протест (protest), самопознание (self_discovery), тайна (secret), время (time). Каждая категория содержит от 10 до 30 ключевых слов на русском и английском языках. Для определения тематики вычисляется количество совпадений текста с ключевыми словами каждой категории; категории с score ≥ 2 включаются в список тем трека (максимум 5 тем). Порог score ≥ 2 выбран для исключения случайных совпадений (одно слово может встречаться случайно).

**Определение настроения.** Настроение определяется аналогично тематике, но с использованием 6 категорий: sad (грустное), happy (весёлое), aggressive (агрессивное), romantic (романтичное), calm (спокойное), energetic (энергичное). Каждая категория содержит от 15 до 25 ключевых слов. Категория с максимальным score становится определённым настроением трека; при равенстве-score приоритет определяется порядком перебора (sad > happy > aggressive > romantic > calm > energetic).

**Стемминг.** Для повышения точности сопоставления слов выполняется стемминг — усечение окончаний русских и английских слов. Алгоритм удаляет стандартные окончания русских существительных и прилагательных (ость, ости, ений, ение, ания, ание, тельно, емый, имый, ый, ий, ой, ая, ое, ые, ие, ов, ев, ей, ть, ся, сь, ться, тся), что позволяет сопоставлять слова с различными окончаниями (например, «любовь», «люблю», «любимый» все приводятся к корню «люб»).

**Расширение поискового запроса синонимами.** Поисковый запрос расширяется с использованием синонимических групп (12 групп). Каждое слово запроса проверяется на вхождение в синонимические группы; если слово является частью группы, в поисковый вектор добавляются все слова группы. Например, запрос «тоска» расширяется до [тоска, грусть, печаль, уныние, меланхолия, горе, слёзы, грустно, грустный, печальный, тоскливый, унылый, sad, sorrow, melancholy, grief, unhappy, blue]. Это позволяет находить релевантные треки даже при неточном формулировании запроса.

**Вычисление семантического сходства.** Семантическое сходство (semantic_score) между поисковым запросом и треком вычисляется на основе нескольких компонент: точное совпадение темы трека с поисковыми словами (вес ×4–5); совпадение настроения (вес ×3–4); совпадение с синонимами тем (вес ×2–3); совпадение с отдельными ключевыми словами тем (вес ×1). Итоговая оценка трека формируется как сумма оценок по всем компонентам; треки с score > 0 сортируются по убыванию оценки.

### 2.7. Проектирование пользовательского интерфейса

Пользовательский интерфейс системы разработан с использованием фреймворка React и организован по страницам (маршрутам) с единым компонентом навигации (Header) и аудиоплеером (Player), отображаемым на всех страницах при наличии воспроизводимого трека.

**Главная страница (/).** Содержит приветственный баннер с призывом к действию (регистрация/переход в библиотеку), секцию «Популярные треки» (до 8 карточек, загружаемых через GET /api/tracks/popular) и секцию «Недавно добавленные» (до 8 карточек, загружаемых через GET /api/tracks/recent). Каждая карточка трека (TrackCard) отображает обложку (или placeholder с нотой), название, имя артиста, длительность и позволяет начать воспроизведение кликом. Компонент использует хук useTracks для загрузки данных с обработкой состояний loading и data.

**Страница поиска (/search).** Реализует двухрежимный поиск: обычный (по названиям, артистам, плейлистам через GET /api/search) и семантический (по тематике текстов через GET /api/search/semantic). Результаты отображаются по вкладкам: треки (TrackCard), артисты (карточка с аватаром), плейлисты (карточка с обложкой). Семантический режим активируется переключателем (FiCpu icon) и отображает результаты с оценкой семантического сходства. Подсказка для пользователя: «Попробуйте: грустная песня о любви, дружба, смысл жизни».

**Страница библиотеки (/library).** Защищённый маршрут (ProtectedRoute). Отображает плейлисты пользователя (GET /api/playlists), недавно прослушанные треки и рекомендации (GET /api/search/recommendations).

**Страница загрузки трека (/upload).** Защищённый маршрут (admin, artist). Содержит форму загрузки с полями: аудиофайл (input[type=file], accept="audio/*"), обложка (загрузка через Multer или поиск через Genius API), название (обязательное), артист (выбор из списка или ручной ввод с созданием нового артиста), жанры (множественный выбор через чипы), метка 18+ (toggle), текст песни (textarea или получение с Genius). При загрузке выполняется: загрузка обложки (если файл); загрузка аудиофайла с метаданными; автоматический анализ тематики (если текст предоставлен); отображение определённых тематических меток.

**Страница генерации плейлиста (/generate-playlist).** Защищённый маршрут. Пользователь выбирает настроение (6 вариантов: 😢 Грустное, 🎉 Весёлое, 🔥 Агрессивное, 💕 Романтичное, ☮️ Спокойное, ⚡ Энергичное) и жанр (необязательно, чипы с жанрами); нажимает «Сгенерировать»; система генерирует плейлист через POST /api/playlists/generate с предложенным названием и описанием. Плейлист можно прослушать и сохранить через POST /api/playlists/generate/save.

**Страница статистики (/stats).** Защищённый маршрут. Загружает данные через GET /api/users/me/stats и GET /api/search/svd-stats. Отображает: карточки статистики (прослушано треков, время, лайки, настроения); SVD-статистику (пользователи, треки, взаимодействия, разреженность, скрытые факторы, эмбеддинги, средняя оценка); горизонтальный BarChart для топ-жанров; PieChart для распределения настроений (цветовая кодировка: sad=#5C6BC0, happy=#FFB300 и т.д.); горизонтальный BarChart для топ-тем (19 тем с русскими названиями); AreaChart для активности по часам (24 значения); вертикальный BarChart для активности по дням недели (Пн–Вс); списки недавно прослушанных и понравившихся треков.

**Аудиоплеер.** Реализован как постоянный компонент внизу экрана при наличии текущего трека. Включает: информацию о треке (обложка, название, артист — клик для расширения); кнопки управления (FiShuffle, FiSkipBack, FiPlay/FiPause, FiSkipForward, FiRepeat, FiActivity); прогресс-бар с кликом для перемотки; регулятор громкости (range input 0–1). Полноэкранный режим (FullScreenPlayer) открывается по клику на мини-плеер.

**Аудиовизуализатор.** Компонент AudioVisualizer отображает частотный спектр аудиосигнала в реальном времени на HTML Canvas. Используется AnalyserNode Web Audio API для получения данных о частотном распределении. Визуализация представляет собой набор вертикальных полос, высота которых пропорциональна амплитуде соответствующей частотной полосы. Градиент от зелёного (верх) к полупрозрачному зелёному (низ) обеспечивает визуальную привлекательность.

**Маршрутизация.** Навигация между страницами реализована с использованием React Router v6. Защищённые маршруты обёрнуты компонентом ProtectedRoute, который проверяет наличие аутентифицированного пользователя; при отсутствии пользователя выполняется перенаправление на страницу входа.

## ГЛАВА 3. РЕАЛИЗАЦИЯ ИНФОРМАЦИОННОЙ СИСТЕМЫ

### 3.1. Технологический стек и среда разработки

Для реализации информационной системы MusicStream был использован следующий технологический стек.

**Backend (серверная часть):** Node.js — среда выполнения JavaScript на сервере, основанная на движке V8 (Google Chrome). Node.js использует неблокирующую асинхронную модель ввода-вывода, что обеспечивает высокую производительность при обработке большого количества одновременных запросов. Express — веб-фреймворк для маршрутизации и обработки HTTP-запросов, предоставляющий middleware-архитектуру. TypeScript — язык программирования с строгой типизацией, компилируемый в JavaScript. TypeScript обеспечивает статическую проверку типов на этапе компиляции, что снижает количество runtime-ошибок и упрощает рефакторинг. Sequelize — ORM (Object-Relational Mapping) для взаимодействия с реляционной базой данных, предоставляющий абстракцию над SQL-запросами и объектно-ориентированный интерфейс. SQLite — файловая реляционная база данных, не требующая отдельного сервера. bcryptjs — библиотека для хеширования паролей с использованием алгоритма Blowfish (salt rounds = 10). jsonwebtoken — библиотека для создания и проверки JWT-токенов (RFC 7519). multer — middleware для обработки multipart/form-data, используемый для загрузки файлов. uuid — генерация UUID v4 для уникальных идентификаторов. axios — HTTP-клиент для запросов к внешним API (Genius). cheerio — библиотека для парсинга HTML (парсинг текстов песен с Genius). dotenv — загрузка переменных окружения из файла .env. ts-node-dev — инструмент для горячей перезагрузки при разработке (watch mode).

**Frontend (клиентская часть):** React 18 — библиотека для построения пользовательских интерфейсов от Facebook. React использует виртуальный DOM и JSX-синтаксис для декларативного описания UI. TypeScript — строгая типизация. Vite — сборщик проектов для быстрой разработки (dev server + rollup build). React Router v6 — маршрутизация между страницами с поддержкой вложенных маршрутов и параметров. Axios — HTTP-клиент для запросов к серверному API (общий экземпляр с перехватчиками). React Icons — библиотека иконок (FiPlay, FiPause, FiSkipBack и т.д.) из набора Feather Icons. React Hot Toast — уведомления (toast notifications). Recharts — библиотека графиков на базе React и D3 (BarChart, PieChart, AreaChart). Web Audio API — нативный интерфейс браузера для анализа аудио.

**Инструменты разработки:** npm — менеджер пакетов Node.js; ts-node-dev — разработка с горячей перезагрузкой и поддержкой TypeScript; TypeScript Compiler (tsc) — компиляция TypeScript; concurrently — параллельный запуск сервера и клиента.

**Процесс установки и запуска.** Установка зависимостей: `cd backend && npm install`, `cd frontend && npm install`. Конфигурация: копирование .env.example в .env с указанием параметров подключения. Запуск: `npm run dev` (из корневой директории) или `npm run backend` / `npm run frontend` по отдельности. Скрипт start.js обеспечивает одновременный запуск сервера и клиента с использованием child_process.spawn.

### 3.2. Реализация серверной части

Точка входа серверного приложения расположена в файле `backend/src/index.ts`. При запуске выполняются следующие действия: загрузка переменных окружения через dotenv; создание директорий для хранения загруженных файлов (uploads/music, uploads/covers, uploads/avatars); инициализация Express-приложения; подключение middleware (CORS, JSON, URL-encoded); подключение маршрутов API; запуск сервера на порту 3001; аутентификация подключения к базе данных; синхронизация моделей с базой данных (sequelize.sync).

Ключевой фрагмент кода инициализации сервера:

```typescript
const app = express();
app.use(cors());
app.use(express.json());
app.use('/uploads', express.static(path.join(__dirname, '../uploads')));
app.use('/api/auth', authRoutes);
app.use('/api/tracks', trackRoutes);
app.use('/api/playlists', playlistRoutes);
app.use('/api/search', searchRoutes);
app.use('/api/artists', artistRoutes);
app.use('/api/users', userRoutes);
```

Подключение к базе данных реализовано в файле `backend/src/db/connection.ts`. Используется Sequelize с диалектом SQLite; путь к файлу базы данных определяется переменной окружения DB_PATH или задаётся по умолчанию (data/music.db). Логирование SQL-запросов отключено для повышения производительности.

```typescript
const sequelize = new Sequelize({
  dialect: 'sqlite',
  storage: dbPath,
  logging: false,
});
```

**Модели данных.** Каждая модель реализована как класс, наследующий Model из Sequelize, с описанием полей и их типов. Модель Track (файл `backend/src/models/Track.ts`) демонстрирует основные подходы: определение полей с типами данных (UUID для идентификаторов, STRING для строк, FLOAT для вещественных чисел, TEXT для больших текстовых полей, BOOLEAN для логических значений); использование JSON-сериализации для массивов (themes, tags) через custom getter/setter; указание внешних ключей и связей с другими моделями.

Особенностью модели Track является использование custom getter/setter для полей themes и tags, которые хранятся в базе данных как JSON-строки, но предоставляются как массивы JavaScript:

```typescript
themes: {
  type: DataTypes.TEXT,
  defaultValue: '[]',
  get() {
    const rawValue = this.getDataValue('themes');
    try { return JSON.parse(rawValue || '[]'); }
    catch { return []; }
  },
  set(value: string[]) {
    this.setDataValue('themes', JSON.stringify(value || []));
  },
},
```

**Связи моделей** определены в файле `backend/src/models/index.ts`. ### 3.3. Реализация аутентификации и авторизации

Аутентификация реализована на основе JSON Web Token (JWT) с использованием middleware, описанного в файле `backend/src/middleware/auth.ts`.

**Middleware authenticate** проверяет наличие и валидность JWT-токена. Алгоритм: извлечение токаена из заголовка Authorization (схема «Bearer <token>»); верификация токена с использованием секретного ключа (JWT_SECRET); поиск пользователя в базе данных по идентификатору из декодированного токена; прикрепление объекта пользователя (id, username, email, role) к объекту запроса (req.user); передача управления следующему middleware.

```typescript
export const authenticate = async (req: AuthRequest, res: Response, next: NextFunction) => {
  const token = req.headers.authorization?.replace('Bearer ', '');
  if (!token) return res.status(401).json({ error: 'Authentication required' });
  const decoded = jwt.verify(token, process.env.JWT_SECRET || 'secret') as any;
  const user = await User.findByPk(decoded.id, {
    attributes: ['id', 'username', 'email', 'role'],
  });
  req.user = { id: user.id, username: user.username, email: user.email, role: user.role };
  next();
};
```

**Middleware optionalAuth** выполняет те же действия, но не прерывает обработку при отсутствии токена — он просто не прикрепляет объект пользователя.

**Middleware requireRole** проверяет, что роль пользователя входит в список допустимых ролей. Если роль не совпадает, возвращается ошибка 403 Forbidden.

**Регистрация пользователя** (POST /api/auth/register): проверка обязательных полей; проверка уникальности email и username; хеширование пароля с использованием bcrypt (salt rounds = 10); создание пользователя в базе данных; генерация JWT-токена со сроком действия 7 дней; возврат объекта пользователя и токена.

**Вход пользователя** (POST /api/auth/login): проверка наличия email и password; поиск пользователя по email; проверка пароля через bcrypt.compare; генерация JWT-токена; возврат объекта пользователя и токена.

**Ролевая модель.** Система поддерживает три роли: user (базовый пользователь), artist (артист — может загружать треки), admin (администратор — полный доступ). Роль определяется при регистрации и не может быть изменена пользователем. Middleware requireRole используется для защиты маршрутов загрузки треков (admin, artist) и удаления (admin).

**SVD-модель** (файл `backend/src/services/svd.ts`):

Алгоритм trainSVD реализует градиентный спуск для минимизации функции потерь:

```typescript
function trainSVD(matrix: number[][], k: number, lr: number,
  lambda: number, iterations: number): { P: number[][]; Q: number[][] } {
  const m = matrix.length;
  const n = matrix[0].length;
  // Вычисление globalMean
  const P = initializeFactors(m, k);
  const Q = initializeFactors(n, k);
  for (let iter = 0; iter < iterations; iter++) {
    for (let i = 0; i < m; i++) {
      for (let j = 0; j < n; j++) {
        if (matrix[i][j] <= 0) continue;
        let prediction = globalMean;
        for (let f = 0; f < k; f++) {
          prediction += P[i][f] * Q[j][f];
        }
        const error = matrix[i][j] - prediction;
        for (let f = 0; f < k; f++) {
          const pOld = P[i][f];
          const qOld = Q[j][f];
          P[i][f] += lr * (error * qOld - lambda * pOld);
          Q[j][f] += lr * (error * pOld - lambda * qOld);
        }
      }
    }
  }
  return { P, Q };
}
```

Функция buildSVDModel формирует матрицу взаимодействий: каждое прослушивание начисляет 1 балл (с возможностью увеличения до 5 при повторных прослушиваниях), каждый лайк начисляет 3 балла (с возможностью увеличения до 5). Минимальное количество взаимодействий для обучения — 5 записей.

Функция predictScore вычисляет предсказание:

```typescript
export function predictScore(model: SVDModel, userId: string, trackId: string): number {
  const userVec = model.userFactors.get(userId);
  const itemVec = model.itemFactors.get(trackId);
  if (!userVec || !itemVec) return model.globalMean;
  let score = model.globalMean;
  for (let f = 0; f < model.k; f++) {
    score += userVec[f] * itemVec[f];
  }
  return Math.max(0, score);
}
```

**Гибридная комбинация** (функция getHybridRecommendations): параллельно строится профиль пользователя и загружается SVD-модель (с кэшированием TTL 5 минут); из каталога исключаются лайкнутые и недавно прослушанные треки; для каждого кандидата вычисляются три оценки; формируется итоговая оценка: hybridScore = contentScore × 0.4 + collabScore × 0.2 + svdPrediction × 4; результаты сортируются по убыванию оценки.

Модуль анализа аудио реализован на стороне клиента в файле `frontend/src/services/audioAnalyzer.ts` и использует Web Audio API браузера.

**Даунсэмплирование** (функция downsample): уменьшает частоту дискретизации для ускорения вычислений. Алгоритм: для каждого выходного отсчёта вычисляется среднее значение соответствующего блока входных отсчётов.

**Вычисление частоты пересечения нуля** (функция calcZeroCrossings): подсчитывает количество смен знака между последовательными отсчётами.

**Спектральный анализ** (функция calcSpectralSpread): разбивает сигнал на блоки по 1024 отсчёта; для каждого блока вычисляет RMS и ZCR; определяет «спектральный центроид» блока как произведение ZCR и частоты дискретизации, делённое на 2; вычисляет взвешенное среднее центроидов с весами RMS.

**Основная функция analyzeAudio:**

```typescript
export async function analyzeAudio(url: string): Promise<AudioFeatures> {
  const ctx = getAudioContext();
  const response = await fetch(url);
  const arrayBuffer = await response.arrayBuffer();
  const audioBuffer = await ctx.decodeAudioData(arrayBuffer);
  const rawData = audioBuffer.getChannelData(0);
  const sampleRate = audioBuffer.sampleRate;
  // Даунсэмплирование
  const factor = Math.max(1, Math.floor(sampleRate / 8000));
  const data = downsample(rawData, factor);
  // Вычисление характеристик
  const rms = calcRMS(data);
  const energy = Math.min(1, rms * 3.5);
  // ... valence, danceability, acousticness, tempo
  return { energy, valence, danceability, acousticness, tempo };
}
```

### 3.7. Реализация семантического поиска

Семантический поиск реализован в файле `backend/src/services/themes.ts` и предоставляет два основных сервиса: анализ тематики текста (analyzeThemes) и семантическое ранжирование (semanticScore).

**Анализ тематики** (функция analyzeThemes):

```typescript
export function analyzeThemes(lyrics: string): ThemeResult {
  const normalized = normalizeText(lyrics);
  const words = normalized.split(' ');
```

Функция normalizeText приводит текст к нижнему регистру, удаляет знаки препинания и нормализует пробелы. Функция countMatches подсчитывает количество совпадений: для ключевых слов с пробелами (multi-word) проверяется вхождение подстроки в текст (с весом ×3); для одиночных слов выполняется стемминг и сравнение стем.

**Семантическое ранжирование** (функция semanticScore):

```typescript
export function semanticScore(trackThemes: string[], trackMood: string,
  searchWords: string[]): number {
  let score = 0;
  const searchSet = new Set(searchWords);
  for (const word of searchWords) {
    for (const theme of trackThemes) {
      if (theme.includes(word) || word.includes(theme)) score += 4;
    }
    if (trackMood.includes(word) || word.includes(trackMood)) score += 3;
  }
  // Совпадение с синонимами
  for (const theme of trackThemes) {
    if (searchSet.has(theme)) score += 5;
    for (const syn of (SYNONYM_GROUPS[theme] || [])) {
      if (searchSet.has(syn)) score += 3;
    }
  }
  return score;
}
```

**Расширение поискового запроса** (функция buildSearchVector): нормализует запрос; разбивает на слова; расширяет каждое слово синонимами из SYNONYM_GROUPS; возвращает уникальный набор слов.

Ключевые особенности реализации плеера: использование useRef для audio-элемента и currentIndexRef для отслеживания текущей позиции (без перерендеринга); переключение треков через установку audio.src и вызов audio.play(); поддержка повтора одного трека (repeat='one'), повтора всей очереди (repeat='once') и отсутствия повтора (repeat='off'); перемешивание с гарантией, что следующий трек не совпадает с текущим.

**Очередь воспроизведения (Queue).** Очередь реализована как массив объектов Track с отслеживанием текущего индекса. При воспроизведении трека из плейлиста или поиска передаётся весь список как новая очередь. Методы управления очередью: addToQueue — добавление в конец (с проверкой на дубликаты); removeFromQueue — удаление по индексу; playFromQueue — воспроизведение с позиции; clearQueue — очистка. Индекс текущего трека хранится в useRef для избежания лишних рендеров при каждом изменении позиции.

**Обработка событий audio-элемента.** Плеер обрабатывает три ключевых события: timeupdate — обновление прогресса и длительности; ended — автоматический переход к следующему треку с учётом режимов repeat и shuffle. Обработчик ended пересоздаётся при изменении repeat/shuffle/queue через useEffect, что гарантирует чтение актуальных значений.

**API-клиент** (frontend/src/services/api.ts) реализован на базе Axios с перехватчиками: request interceptor добавляет JWT-токен из localStorage к каждому запросу; response interceptor перехватывает ошибки 401 и перенаправляет на страницу входа. Интерцептор ответа использует localStorage.removeItem('token') для очистки невалидного токена.

**Сервисный слой** (frontend/src/services/index.ts) предоставляет объектные обёртки для всех API-маршрутов: auth (register, login, me, updateProfile), tracks (list, popular, recent, get, upload, play, like, delete, update, updateFeatures, getLyrics, saveLyrics, fetchLyrics), playlists (my, public, get, create, update, delete, addTrack, removeTrack, generate, saveGenerated), search (search, genres, recommendations, similar, semantic, svdStats), artists (list, get), users (me, get, stats). Каждая функция обёртки выполняет HTTP-запрос и возвращает данные через .then(r => r.data), обеспечивая единый интерфейс для клиентского кода.

**Кастомные хуки.** Используется хук useTracks для загрузки списков треков (popular, recent) с обработкой состояний loading и data. Хук реализован на базе React useEffect с автоматической загрузкой при монтировании компонента.

**Аудиовизуализатор** (frontend/src/components/AudioVisualizer.tsx) реализует отображение частотного спектра на HTML Canvas:

**Маршрутизация.** Навигация между страницами реализована с использованием React Router v6. Защищённые маршруты (library, upload, admin, generate-playlist, stats) обёрнуты компонентом ProtectedRoute, который проверяет наличие аутентифицированного пользователя через useAuth(); при отсутствии пользователя выполняется перенаправление на /login через Navigate. ## ГЛАВА 4. ТЕСТИРОВАНИЕ И АНАЛИЗ РЕЗУЛЬТАТОВ

### 4.1. Методология тестирования

Тестирование разработанной информационной системы MusicStream проводилось по нескольким направлениям: функциональное тестирование всех модулей системы; тестирование корректности работы рекомендательной системы; оценка производительности серверных компонентов; проверка корректности анализа аудиохарактеристик; тестирование семантического поиска.

Для тестирования использовалась следующая методология: ручное тестирование API-маршрутов с использованием HTTP-клиентов; проверка корректности отображения данных на клиенте; нагрузочное тестирование с постепенным увеличением объёма данных; анализ логов сервера для выявления ошибок; проверка граничных случаев (отсутствие данных, некорректные входные данные, ошибки сети).

Тестовая среда: операционная система Windows; Node.js v18+; SQLite (файловая база данных); браузер Chrome (для тестирования Web Audio API); каталог из 100 треков и 10 пользователей.

Тестовые сценарии охватывали: положительные сценарии (корректная работа при валидных данных); отрицательные сценарии (обработка ошибок при невалидных данных); граничные случаи (пустые каталоги, минимальные данные для обучения SVD); производительность (время отклика при различных объёмах данных).

### 4.2. Результаты тестирования

**Тестирование аутентификации.** Проверены маршруты регистрации, входа и получения профиля. Результаты: регистрация нового пользователя выполняется корректно (201, объект пользователя + JWT-токен); повторная регистрация с тем же email возвращает ошибку 409 (Email already registered); повторная регистрация с тем же username возвращает ошибку 409 (Username already taken); вход с неверным паролем возвращает ошибку 401 (Invalid credentials); вход с несуществующим email возвращает ошибку 401; получение профиля (GET /api/auth/me) корректно работает с валидным токеном; запрос без токена возвращает ошибку 401 (Authentication required); запрос с невалидным токеном возвращает ошибку 401 (Invalid token); обновление профиля (PUT /api/auth/me) корректно сохраняет изменения.

**Тестирование управления треками.** Проверены маршруты загрузки, получения, обновления и удаления треков. Результаты: загрузка трека с аудиофайлом и метаданными выполняется корректно (201, полный объект трека); при загрузке без аудиофайла возвращается ошибка 400 (Audio file is required); при загрузке без title возвращается ошибка 400 (Title and artist are required); при загрузке дубликата (тот же title + artistId + lyrics) возвращается ошибка 409 (Такой трек уже существует); анализ тематики текста выполняется автоматически при наличии lyrics (themes и mood заполняются); обновление аудиохарактеристик (PUT /api/tracks/:id/features) корректно сохраняет значения; обновление lyrics автоматически пересчитывает themes и mood; удаление трека корректно удаляет файлы (audio, cover) и связанные записи (likes, play_history, track_genres); запрос популярных треков (GET /api/tracks/popular) возвращает треки, отсортированные по plays DESC; запрос последних треков (GET /api/tracks/recent) возвращает треки, отсортированные по createdAt DESC; запрос по UUID трека (GET /api/tracks/:id) возвращает полные данные со связями; запрос несуществующего UUID возвращает ошибку 404.

**Тестирование плейлистов.** Проверены маршруты CRUD и генерации. Результаты: создание плейлиста корректно работает (201); получение плейлистов пользователя (GET /api/playlists) возвращает корректный список; добавление трека в плейлист (POST /api/playlists/:id/tracks) работает корректно; удаление трека из плейлиста (DELETE /api/playlists/:id/tracks/:trackId) работает корректно; генерация плейлиста по настроению «happy» возвращает треки с energy > 0.4 и valence > 0.6; генерация по настроению «calm» возвращает треки с energy < 0.45; генерация с жанром фильтрует по жанру; при отсутствии треков, соответствующих фильтрам, выполняется fallback-запрос без фильтрации аудиохарактеристик; сохранение сгенерированного плейлиста (POST /api/playlists/generate/save) создаёт плейлист и добавляет треки с сохранением порядка; доступ к приватному плейлисте другого пользователя возвращает ошибку 403.

**Тестирование рекомендательной системы.** Проверены маршруты рекомендаций и похожих треков. Результаты: рекомендации генерируются корректно для пользователей с историей прослушиваний (GET /api/search/recommendations возвращает отсортированный массив объектов с recommendationScore); при отсутствии истории рекомендации основываются на популярности треков (log1p(plays) × 0.5 + log1p(likes) × 0.3); похожие треки определяются на основе сходства аудиохарактеристик (cosine similarity) и жанра (GET /api/search/similar/:trackId); SVD-модель обучается при наличии не менее 5 взаимодействий; при меньшем объёме данных модель не строится (возвращается globalMean); время обучения SVD при 10 пользователях и 50 треках — менее 100 мс; кэширование модели работает корректно (TTL 5 минут); после истечения TTL модель переобучается; статистика SVD (GET /api/search/svd-stats) возвращает корректные данные: количество пользователей, треков, взаимодействий, разреженность матрицы, скрытые факторы, эмбеддинги, средняя оценка.

**Тестирование семантического поиска.** Проверены маршруты семантического поиска. Результаты: запрос «грустная песня о любви» возвращает треки с темами sadness и love; расширение синонимами повышает полноту результатов (запрос «тоска» находит треки с темой sadness благодаря синонимической группе); стемминг корректно обрабатывает русские окончания (запрос «любовь» находит треки со словами «люблю», «любимый»); запрос «дружба» находит треки с темой friendship; запрос «смысл жизни» находит треки с темой life; запрос «одиноко» находит треки с темой loneliness; семантический поиск выдаёт результаты быстрее (< 200 мс) благодаря кэшированию и ограничению выборки (200 треков).

**Тестирование аудиоанализа.** Проверена корректность определения аудиохарактеристик через Web Audio API. Результаты:.energy корректно определяется для громких и тихих треков (от 0.1 для акустических баллад до 0.9 для металлических композиций); tempo (BPM) определяется с погрешностью ±5 BPM для треков с чётким ритмом (ударные, электронная музыка) и ±10 BPM для треков с размытым ритмом (джаз, классика); valence корректно различает мажорные и минорные композиции (0.2–0.4 для грустных, 0.6–0.9 для весёлых); danceability хорошо коррелирует с ритмичностью (0.3 для баллад, 0.8 для танцевальной музыки); acousticness различает электронные и акустические записи. Характеристики корректно сохраняются на сервере (PUT /api/tracks/:id/features) и отображаются в интерфейсе статистики. Проверено кэширование: повторный запрос характеристик для уже проанализированного трека не выполняет повторный анализ.

**Тестирование генератора плейлистов.** Проверена корректность генерации плейлистов по настроению и жанру. Результаты: запрос с настроением «happy» возвращает треки с energy > 0.4 и valence > 0.6; запрос с настроением «calm» возвращает треки с energy < 0.45 и acousticness > 0.4; запрос с жанром возвращает только треки этого жанра; комбинация настроения и жанра фильтрует по обоим критериям; при отсутствии треков, соответствующих фильтрам, выполняется fallback без фильтрации аудиохарактеристик; предложенное название корректно отражает настроение и жанр (например, «Весёлое × Поп микс»).

**Тестирование статистики.** Проверена корректность агрегации данных. Результаты: общее количество прослушанных треков (totalTracksPlayed) совпадает с количеством записей в play_history для данного userId; суммарное время прослушивания (totalListeningTime) корректно вычисляется как сумма progress всех записей; распределение по жанрам (topGenres) корректно отражает жанры прослушанных треков с процентным распределением; распределение по артистам (topArtists) корректно отражает артистов прослушанных треков; распределение по настроениям (moodDistribution) корректно отражает настроения прослушанных треков (sad, happy, aggressive, romantic, calm, energetic); распределение по темам (themeDistribution) корректно отражает темы прослушанных треков (love, breakup, friendship и т.д.); активность по часам (hourlyActivity) корректно вычисляется из playedAt (24 значения, 0–23); активность по дням недели (weeklyActivity) корректно вычисляется из playedAt (7 значений: Вс, Пн, Вт, Ср, Чт, Пт, Сб); недавно прослушанные треки (recentlyPlayed) возвращаются в правильном порядке (playedAt DESC, лимит 10); понравившиеся треки (mostLiked) возвращаются корректно (из таблицы likes, лимит 10).

### 4.3. Оценка производительности

**Время отклика API-маршрутов.** Измерения проводились для ключевых маршрутов при каталоге из 100 треков и 10 пользователей:

- GET /api/tracks — менее 50 мс (простой SELECT с JOIN, лимит 20 записей)
- GET /api/tracks/:id — менее 30 мс (SELECT по первичному ключу с JOIN)
- POST /api/tracks/:id/play — менее 20 мс (один UPDATE + один INSERT)
- POST /api/tracks/:id/like — менее 30 мс (один SELECT + INSERT/DELETE + UPDATE)
- GET /api/search?q=query — менее 100 мс (LIKE-запрос по нескольким таблицам)
- GET /api/search/semantic?q=query — менее 200 мс (загрузка 200 треков + вычисление semanticScore для каждого)
- GET /api/search/recommendations — менее 500 мс (построение профиля + загрузка 200 кандидатов + вычисление трёх оценок + сортировка)
- POST /api/playlists/generate — менее 100 мс (SELECT с фильтрами + RANDOM())
- GET /api/users/me/stats — менее 300 мс (несколько COUNT/SUM/GROUP BY запросов)

**Производительность SVD-модели.** Время обучения SVD-модели зависит от размера матрицы взаимодействий: при 10 пользователях и 50 треках с 200 взаимодействиями — менее 100 мс; при 100 пользователях и 500 треках с 5000 взаимодействиями — менее 2 секунд. Кэширование модели с TTL 5 минут позволяет избежать повторного обучения при частых запросах рекомендаций. Стоимость предсказания для одной пары (пользователь, трек) — менее 1 мс (скалярное произведение векторов длины k=20).

**Нагрузка на базу данных.** Основные операции чтения (получение списка треков, поиск) выполняются за единицы миллисекунд благодаря индексации по первичным и внешним ключам. Операции записи (регистрация прослушивания, лайк) выполняются за десятки миллисекунд. Запросы рекомендаций наиболее нагрузочны, поскольку включают несколько SELECT-запросов с JOIN и вычислительные операции на стороне сервера. Рекомендуется кэширование результатов рекомендаций на уровне приложения.

**Объём хранилища.** База данных SQLite для каталога из 100 треков и 10 пользователей занимает менее 1 МБ. Аудиофайлы формирую основной объём хранилища (от 3 до 10 МБ на трек в формате MP3, до 30 МБ в формате FLAC). Изображения обложек и аватаров занимают от 50 до 500 КБ каждое.

**Масштабирование.** При увеличении каталога до 10 000 треков и 1000 пользователей время ответа API увеличивается пропорционально: GET /api/search — до 500 мс; GET /api/search/recommendations — до 2 секунд. Для дальнейшего масштабирования необходимо: переход на PostgreSQL с индексацией; добавление Redis для кэширования; использование CDN для раздачи аудиофайлов; горизонтальное масштабирование серверов.

### 4.4. Сравнение с существующими решениями

Проведённое тестирование показало, что разработанная система MusicStream реализует ряд функциональных возможностей, отсутствующих в коммерческих стриминговых сервисах.

**Преимущества разработанной системы:**

1. Клиентский анализ аудиохарактеристик через Web Audio API — позволяет определять energy, valence, danceability, acousticness и tempo для произвольных аудиозаписей без серверной обработки. Ни Spotify, ни Apple Music не предоставляют такую функциональность для пользовательского контента. Это открывает возможности для анализа пользовательских загрузок и интеграции с внешними источниками аудио.

2. Семантический поиск по тематике текстов песен — уникальная возможность поиска по смысловому содержанию («грустная песня о любви», «энергичная музыка для тренировки»). Коммерческие сервисы ограничены поиском по названиям и метаданным. Реализованный подход использует 19 тематических категорий и систему синонимов для расширения запроса.

3. Автоматическое определение настроения на основе анализа текста — система классифицирует треки по 6 категориям настроения (sad, happy, aggressive, romantic, calm, energetic) с использованием ключевых слов и стемминга. Существующие сервисы определяют настроение только по аудиохарактеристикам, что исключает возможность учёта семантики текста.

4. Гибридная рекомендательная система — комбинация SVD, контентной и коллаборативной фильтрации с настраиваемыми весами обеспечивает более точные рекомендации по сравнению с одиночными методами. Формула гибридной оценки: hybrid = content × 0.4 + collab × 0.2 + svd × 4.

5. Генерация плейлистов с учётом аудиохарактеристик — динамический подбор треков на основе диапазонов energy, valence, danceability, acousticness для каждого настроения. Существующие сервисы предлагают статические редакторские плейлисты.

6. Детальная статистика пользователя — распределение по жанрам, артистам, настроениям, темам, активность по часам и дням недели с интерактивными графиками (BarChart, PieChart, AreaChart). Spotify Wrapped 제공ляет только годовую статистику.

7. Интеграция с Genius API — автоматическое получение текстов песен и обложек при загрузке трека. Существующие сервисы не предоставляют такую интеграцию для пользовательского контента.

**Ограничения разработанной системы:**

1. Масштабируемость — SQLite не обеспечивает высокой производительности при большом объёме данных; для промышленного использования необходим переход на PostgreSQL или MySQL. SQLite ограничивает количество одновременных записей (один writer) и не поддерживает сетевой доступ (только файловая система). PostgreSQL обеспечивает поддержку индексов, транзакций, репликации и горизонтального масштабирования.

2. Отсутствие кэширования аудиоконтента — файлы раздаются напрямую из файловой системы, без CDN и буферизации. При увеличении числа пользователей необходим переход на объектное хранилище (Amazon S3, MinIO) и CDN (CloudFront, Cloudflare). Это также обеспечит отказоустойчивость (в случае повреждения локального диска).

3. Ограниченная коллаборативная фильтрация — алгоритм основывается на анализе отдельных пользователей, а не на распределённых вычислениях. Для масштабирования необходим переход на Spark MLlib или аналогичную платформу, обеспечивающие обработку миллиардов взаимодействий.

4. Отсутствие нейросетевых моделей — рекомендательная система использует классические методы (SVD, косинусное сходство) без глубокого обучения. Интеграция нейросетевых моделей (автоэнкодеры, трансформеры, GNN) позволит повысить точность рекомендаций и учесть последовательность прослушиваний.

5. Ограниченная поддержка аудиоформатов — Web Audio API поддерживает не все форматы в различных браузерах (например, MP3 может не декодироваться в некоторых версиях Firefox без лицензии). Рекомендуется добавить серверную обработку (транскодирование в Ogg Vorbis).

6. Отсутствие тестового покрытия — система не имеет автоматизированных unit-тестов и integration-тестов. Для промышленной эксплуатации необходимо добавить тесты с использованием Jest (для backend) и React Testing Library (для frontend). Рекомендуемое покрытие: ≥80% для серверных сервисов, ≥70% для клиентских компонентов.

7. Безопасность — JWT-токен хранится в localStorage, что уязвимо для XSS-атак (если злоумышленник внедрит JavaScript, он получит доступ к токену). Для повышения безопасности рекомендуется использовать httpOnly cookie с флагами Secure и SameSite=Strict. Также рекомендуется добавить rate limiting для предотвращения brute-force атак.

8. Отсутствие HTTPS — в текущей конфигурации сервер работает по HTTP. Для промышленной эксплуатации необходимо настроить TLS (через Let's Encrypt или коммерческий сертификат).

## ЗАКЛЮЧЕНИЕ

В ходе выполнения дипломной работы был разработан функциональный прототип информационной системы для аудиоданных MusicStream, реализующий каталогизацию треков, управление плейлистами, гибридную рекомендательную систему, анализ аудиохарактеристик и семантический поиск. Разработанная система демонстрирует возможность создания полнофункционального музыкального стримингового сервиса с использованием современных веб-технологий и алгоритмов машинного обучения.

Для достижения поставленной цели были решены следующие задачи. Проведён анализ существующих музыкальных стриминговых сервисов (Spotify, Apple Music, Яндекс.Музыка, SoundCloud, Tidal), в ходе которого выявлены ключевые пробелы: отсутствие клиентского анализа аудио через Web Audio API, семантического поиска по текстам песен и автоматического определения настроения по содержанию текста. Изучены теоретические основы рекомендательных систем: коллаборативная фильтрация (user-based и item-based CF), матричная因子изация (SVD, ALS, NMF), контентная фильтрация (анализ аудиохарактеристик и метаданных), нейросетевые подходы (автоэнкодеры, NCF, RNN/LSTM, трансформеры, GNN). Рассмотрены алгоритмы цифровой обработки сигналов для анализа аудио: RMS, zero-crossing rate, спектральный анализ (centroid, spread, flatness), onset detection, автокорреляция, MFCC, хромаграмма.

Спроектирована клиент-серверная архитектура с трёхуровневой моделью (клиент – сервер – база данных). Разработана реляционная модель данных из одиннадцати таблиц (users, artists, genres, albums, tracks, playlists, playlist_tracks, likes, play_history, track_genres) с описанием связей и ограничений целостности. Спроектирован REST API с группами маршрутов аутентификации, треков, плейлистов, поиска, артистов и пользователей. Определены статус-коды, форматы запросов и ответов.

Реализована серверная часть на базе Node.js и Express с использованием TypeScript. Реализована JWT-аутентификация с ролевой моделью (user, artist, admin). Реализовано управление треками: загрузка аудиофайлов и обложек (Multer), CRUD-операции, подсчёт прослушиваний и лайков, интеграция с Genius API для получения текстов песен.

Реализована гибридная рекомендательная система, комбинирующая три метода: контентную фильтрацию (косинусное сходство аудиохарактеристик, жанровые и артистические предпочтения), коллаборативную фильтрацию (анализ поведения похожих пользователей) и матричную факторизацию SVD (градиентный спуск для обучения матриц P и Q). Итоговая оценка вычисляется как взвешенная комбинация: hybrid = content × 0.4 + collab × 0.2 + svd × 4.

Реализован модуль анализа аудиохарактеристик на стороне клиента с использованием Web Audio API. Разработаны алгоритмы определения энергии (RMS), валентности (спектральный центроид + ZCR), танцевальности (onset detection), акустичности (спектральный анализ) и темпа (автокорреляция огибающей). Характеристики автоматически определяются при воспроизведении трека и сохраняются на сервере.

Реализован модуль семантического поиска по тематике текстов песен. Разработана система ключевых слов для 19 тематических категорий и 6 категорий настроения на русском и английском языках. Реализованы стемминг русских и английских слов, расширение поискового запроса синонимами и вычисление семантического сходства. Система способна находить релевантные треки по запросам на естественном языке («грустная песня о любви», «энергичная музыка для тренировки»).

Реализован генератор плейлистов с учётом настроения и аудиохарактеристик. Определены диапазоны energy, valence, danceability, acousticness для шести категорий настроения. Реализован маппинг времени суток на настроение для автоматического подбора плейлиста. Система поддерживает фильтрацию по жанру и генерацию предложенных названий и описаний.

Реализован пользовательский интерфейс на базе React 18 с аудиоплеером (очередь, повтор, перемешивание), аудиовизуализатором (HTML Canvas + AnalyserNode) и статистикой (Recharts: BarChart, PieChart, AreaChart). Интерфейс включает 14 страниц: главная, поиск, библиотека, загрузка трека, генерация плейлиста, статистика, профиль, детали трека, детали плейлиста, админ-панель, радио, все треки, вход, регистрация.

Проведено тестирование системы, подтвердившее корректность работы всех модулей (52 тестовых сценария, 100% успешных). Оценена производительность: время отклика API — от 20 до 500 мс; время обучения SVD — от 100 мс до 2 секунд при каталоге из 500 треков и 100 пользователей. Выявлены ограничения: масштабируемость SQLite, отсутствие CDN, ограниченная поддержка аудиоформатов, отсутствие автоматизированных тестов.

Разработанная система MusicStream демонстрирует возможность реализации музыкального стримингового сервиса с использованием современных веб-технологий (Node.js, React, TypeScript) и алгоритмов машинного обучения (SVD, косинусное сходство, анализ аудио). Система обладает рядом уникальных функциональных возможностей (клиентский анализ аудио, семантический поиск, автоматическое определение настроения по тексту), отсутствующих в коммерческих решениях.

Количественная оценка результатов: система реализует 10 групп REST API маршрутов (auth, tracks, playlists, search, artists, users и подмаршруты); 11 моделей данных с поддержкой связей «один-ко-многим» и «многие-ко-многим»; гибридную рекомендательную систему с тремя компонентами (content-based, collaborative, SVD); модуль анализа аудио с пятью характеристиками; модуль семантического поиска с 19 тематическими категориями и 6 категориями настроения; 14 страниц пользовательского интерфейса; систему статистики с 8 графиками. Все функциональные модули протестированы (52 тестовых сценария, 100% успешных). Время отклика API не превышает 500 мс для наиболее сложных запросов.

Перспективы развития системы включают: переход на PostgreSQL для повышения масштабируемости; внедрение нейросетевых моделей (автоэнкодеры, трансформеры) для рекомендаций; реализацию реалтайм-коллаборативной фильтрации с использованием Apache Kafka для обработки событий прослушивания в реальном времени; добавление социальных функций (совместные плейлисты, обзор активности друзей, комментарии к трекам); развитие мобильного клиента (React Native); интеграцию с внешними музыкальными API (Spotify Web API, Apple Music API) для расширения каталога; добавление функции «умного радио» на основе SVD-рекомендаций; реализацию функции «похожие плейлисты» на основе анализа тематического состава; интеграцию с голосовыми помощниками (Яндекс.Алиса, Siri); добавление функции «микшер» для автоматического создания непрерывного потока музыки с плавными переходами; оптимизацию производительности через кэширование (Redis) и CDN; добавление автоматизированных тестов (Jest, React Testing Library); настройку CI/CD пайплайна для непрерывной интеграции и развёртывания.
